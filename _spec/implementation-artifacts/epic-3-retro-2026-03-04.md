## What Went Well

**Full validation pipeline delivered end-to-end.** Epic 3 completes the core quality loop: V6 invariant checks → V3 reflexion validation → pipeline routing → retry with feedback injection → halt report on exhaustion. The product now automatically validates and retries agent output — the core safety mechanism works.

**Clean layered architecture validated the design.** The 4-story progression (V6 → V3 → Pipeline → Validate Node) built each layer independently with its own tests, then composed them cleanly. Story 3.4 wired the pipeline into the graph without modifying any validation module — consuming only the `run_validation_pipeline()` API. Package dependency DAG held perfectly.

**V6 short-circuit design proved sound.** When V6 invariant checks fail, the pipeline skips V3 reflexion entirely — saving API calls and tokens. This is a real cost optimization that works as designed. Pipeline tests verify V3 is never called on V6 failure.

**100% follow-through on Epic 2 retro actions.** Every action item from the Epic 2 retrospective was addressed. The shared lint command (carried from Epic 1!) was finally implemented. This is the first epic with complete retro action follow-through.

**Pitfalls block continues to eliminate recurring issues.** Third consecutive epic with zero recurring patterns from previous epics. The "story spec is the agent's memory" principle is fully proven and compounding in value.

**Test infrastructure scaled well.** `MockSDKClient` from Epic 2 was extended for multi-call sequences (Story 3.2/3.4 retry testing) without breaking existing tests. Counter-based mock pattern (`call_count` + `nonlocal`) became the standard for retry integration tests.

**Story 3.4 was the most complex story yet — delivered cleanly.** 20 ACs, replaced a placeholder node, modified 3 modules, added retry loop + halt report + feedback injection + budget tracking, 17 new tests + 3 updated, across engine and agent packages. Only 1 code-review fix needed (halt-report retry count off-by-one).

**Review finding severity continued to trend downward.** Story 3.4 had only 1 finding (auto-fixed by code-review workflow) vs. Story 3.1's 7 findings. Team learning curve within the epic continued the trend from Epic 2.

## What Could Be Improved

**Quality gate claim mismatches persist (2/4 stories).** Story 3.2 claimed "zero lint violations" while `ruff check .` was failing (pre-existing issues in `invoker.py`). Story 3.3 claimed "full suite passes" while tests were broken by missing `tests/__init__.py`. Root cause: dev agents run quality gates in isolation and don't always catch issues originating in other modules. This was HIGH findings in both stories.

**File list discrepancies continue (3/4 stories).** Stories 3.1, 3.2, and 3.3 all had review findings where the Dev Agent Record file list didn't match actual git changes. This is the same pattern from Epic 2 (4/7 stories). The dev agent still has no mechanism to reconcile its file list with git state. The pattern is now 7/11 stories across two epics — it's systemic.

**Pre-existing ruff violation cleaned up (2026-03-04).** The `UP037` violation in `agent/invoker.py:46` (quoted return type annotation) has been present since Epic 2, Story 2.5. Fixed post-retro: removed quotes from `"Path"` annotation — safe because `from __future__ import annotations` is present. `ruff check .` now reports zero violations.

**Halt-report retry count off-by-one (Story 3.4).** The initial implementation used `state.retry_count` (pre-increment) instead of `state.retry_count + 1` (post-increment) in the halt report for V3 max-retry escalation. Caught and auto-fixed by code-review workflow, with regression test added. Root cause: state mutations happen in sequence, and the off-by-one was between "increment count" and "generate report" steps — ordering sensitivity.

**Review finding volume remains significant.** 17 findings across 4 stories (avg 4.25/story) is comparable to Epic 2 (26 across 7 stories, avg 3.7/story). While severity is trending down, the volume suggests dev agents still produce implementations that need non-trivial review corrections.

## Key Insights

1. **Retro action follow-through works when actions are embedded in the process.** The pitfalls block, integration test boilerplate, and self-contained AC audit all work because they're enforced at story creation time — part of the workflow, not afterthoughts. The shared lint command worked because it was a one-time task with clear ownership.

2. **File list reconciliation is a tooling gap, not a process gap.** 7/11 stories across Epics 2–3 have this issue. No amount of "add to pitfalls" will fix it because the dev agent literally doesn't have a mechanism to compare its output list against `git status`. This needs a tooling solution (post-story git diff audit step).

3. **Quality gate claims need automated verification.** Dev agents self-report quality gate results, and they sometimes report success when the gate is actually failing elsewhere. The code-review workflow catches this, but the fix should be upstream: run quality gates as a separate automated step, not rely on dev agent self-reporting.

4. **Layered module design enables clean story decomposition.** Epic 3's 4-story architecture (V6 module → V3 module → Pipeline module → Graph integration) meant each story was independently testable with no cross-story test breakage for the first 3 stories. Story 3.4's integration was clean because it consumed stable APIs.

5. **Off-by-one bugs in state mutation sequences are a predictable pattern.** When a node performs "increment counter" + "use counter in output" + "set status", the ordering matters. Story specs should call out the expected value at each step explicitly when state mutations cascade.

## Key Technical Patterns Established in Epic 3

1. **V6 check registry pattern** — `V6Check` Protocol with `_CHECK_REGISTRY: list[V6Check]` for extensible deterministic checks
2. **V6 short-circuit** — V6 fails → skip V3 (no API call, zero tokens), pipeline returns `FAIL_V6`
3. **Reflexion prompt construction** — Story ACs extracted, sent as separate stateless SDK invocation, parsed for per-AC pass/fail verdicts
4. **Feedback injection on retry** — `ReflexionFeedback.feedback_per_criterion` appended to next agent prompt as `## Previous Validation Feedback` section
5. **`PipelineOutcome` StrEnum routing** — `PASS` / `FAIL_V3` / `FAIL_V6` maps to `TaskState.SUCCESS` / `RETRY` / `ESCALATED`
6. **Counter-based mock pattern** — `nonlocal call_count` in monkeypatched async functions for retry integration tests
7. **Halt report generation** — Structured markdown with story ID, failing ACs, retry history table, truncated agent output, suggested fixes
8. **`retry_history: list[PipelineResult]`** — Accumulated validation results across retry attempts, never overwritten
9. **Validation checkpoint writes** — Each validation result serialized to run directory on state transitions
10. **`model_copy(update={...})` for budget updates** — Consistent pattern from Epic 2 continued for adding validation costs to BudgetState
