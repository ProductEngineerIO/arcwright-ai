## What Went Well

**Story specs reached peak maturity.** Every Epic 5 story included Known Pitfalls (14-17 items each), Existing Code to Reuse (complete import paths and function signatures), Relationship to Other Stories (dependency chain documentation), and Testing Patterns (exact `monkeypatch.setattr` strings). Story 5.1's "all tasks completed in a single implementation pass" validates that spec investment directly reduces implementation friction.

**HaltController consolidation was architecturally clean.** Story 5.2 extracted 6 nearly-identical exception handlers from `_dispatch_epic_async()` into a proper `HaltController` class in `cli/halt.py` (698 lines). Two entry points — `handle_halt()` for exception-based halts and `handle_graph_halt()` for non-SUCCESS graph results — cover all failure modes. The class stores common halt context (`project_root`, `run_id`, `epic_spec`) in the constructor, avoiding parameter repetition across calls.

**Budget carry-forward threaded correctly across 5 stories.** The architectural through-line of Epic 5: Story 5.1 introduced budget carry-forward between stories in the dispatch loop. Story 5.2 recorded budget state at halt time. Story 5.3 deserialized budget from `run.yaml` for resume. Story 5.4 included budget in halt reports. Story 5.5 displayed it in status output. The frozen `BudgetState` Pydantic model prevented accidental mutation — `model_copy(update={...})` was the only way to propagate budget changes, forcing deliberate handling at every stage.

**Spec-before-implementation enforced for all 5 stories.** Action 6 from the Epic 4 retro — "no implementation without a story spec" — held firm. All 5 stories had their spec files created via the `create-story` workflow before the dev agent began implementation. This reversed the Story 4.4 anti-pattern where the spec was written after the code.

**Epic sequencing was clean and intentional.** Each story built directly on its predecessor: dispatch loop (5.1) → halt handling (5.2) → resume from halt (5.3) → artifact enrichment (5.4) → status visibility (5.5). No story had to work around missing infrastructure or parallel dependencies. The point distribution naturally tapered: two 5-point foundation stories, one 5-point dependent story, then two 3-point refinement stories.

**122 tests added — highest per-point ratio.** At 5.8 tests per story point, Epic 5 was the most test-dense epic. Key test modules: `test_dispatch.py` (43 tests), `test_summary.py` (49 tests), `test_halt.py` (26 tests, all new), `test_status.py` (13 tests, all new).

## What Could Be Improved

**dispatch.py grew to 894 lines with mixed concerns.** The file started Epic 5 as the basic Story 2.7 dispatch with ~200 lines of substance. Over 5 stories, it accumulated: epic dispatch with ProjectState, confirmation display, budget carry-forward, resume logic, resume confirmation, run-finding, budget reconstruction from YAML — all alongside the original single-story dispatch. The resume branch alone (~200 lines) could be its own module.

**Every story required AI review follow-ups.** Story 5.1: ProjectState init, halt reason output, exception mapping. Story 5.2: SDK error class coverage, halt-reason taxonomy, NFR2 guard parity. Story 5.4: line-based truncation. Story 5.5: elapsed time datetime parsing, stderr assertion in tests. While the review process caught these effectively, the pattern suggests AC precision could be higher — edge cases are consistently missed in first-pass implementations.

**Story 5.2 spec file status not synced.** The story file `Status:` header still reads "review" while `sprint-status.yaml` correctly shows "done". This is a process gap where the sprint status tracking file was updated but the individual story spec file was forgotten.

**Action 4 (automated quality gates) is now 4 epics old.** First raised in Epic 2's retro, re-raised in Epic 3 and 4, and still deferred. The recurring review findings pattern (every story has follow-ups) directly supports the need for automated quality gate verification in the code-review workflow. The manual process works but costs review cycles.

## Key Insights

1. **Spec quality is the highest-leverage investment.** Story dev notes with Known Pitfalls, Existing Code to Reuse, and Testing Patterns enabled single-pass implementations. The ROI is clear: 30 minutes of extra spec writing saves hours of implementation debugging and review rework.

2. **Frozen Pydantic models are a correctness forcing function.** `BudgetState` carry-forward threaded correctly across 5 stories without a single mutation bug because immutability made incorrect usage impossible. This pattern should extend to any cross-story state model.

3. **Epic sequencing reduces integration complexity.** Each story building on exactly one predecessor eliminated the "integrate 6 prior stories" problem (cf. Story 2.7 in Epic 2). Linear dependency chains are easier to reason about than DAG-shaped dependencies.

4. **dispatch.py is a single-responsibility violation.** File size is a symptom — the root cause is that dispatch, resume, and orchestration live in one module. The pre-Epic 6 refactoring (extract `resume.py`) will establish the pattern for Epic 6's SCM additions.

## Key Technical Patterns Established in Epic 5

1. **HaltController pattern** — centralized halt orchestration with two entry points (`handle_halt` for exceptions, `handle_graph_halt` for graph states). Constructor stores immutable context; methods coordinate artifact writes, provenance flush, status updates, CLI output, and exit code determination.

2. **Budget carry-forward** — `BudgetState` (frozen Pydantic model) propagated through `model_copy(update={...})`. Terminal state from story N becomes initial state for story N+1. On halt, budget serialized to `run.yaml`. On resume, budget deserialized and rehydrated as `BudgetState`. `Decimal(str(value))` prevents float-to-Decimal precision loss.

3. **Resume creates new run** — original halted run's artifacts are read-only (immutable forensic record). Resume reads the halted `run.yaml`, filters completed stories, generates a new `run_id`, creates a new run directory, and executes only remaining stories. Budget carries forward; provenance continues in the new run.

4. **Previous run summary chaining** — `write_halt_report()` and `write_success_summary()` accept `previous_run_id` parameter. When provided, the function reads the previous run's `summary.md` and prepends it under a "Previous Run Report" section. This creates a chronological audit trail across halt/resume sequences.

5. **AC ID extraction from validation history** — `_extract_failing_ac_ids()` parses "V3: ACs 1, 3" patterns from validation history entries. `_extract_failing_ac_ids_from_state()` extracts from `StoryState.retry_history[].feedback.unmet_criteria`. Both feed into the halt report's "Failing ACs" field.

6. **CLI async wrapping pattern** — sync Typer command functions wrap async domain logic via `asyncio.run(_async_impl(...))`. Established in `dispatch_command` → `_dispatch_epic_async()`, extended in `status_command` → `_status_async()`.

7. **Status/cost display** — `status_command` categorizes stories by terminal state (`success`/`completed`/`done` vs `failed`/`halted`/`escalated` vs pending). Budget display formats `invocation_count`, `total_tokens`, `estimated_cost` inline without importing private `_format_budget_field` from output package.
