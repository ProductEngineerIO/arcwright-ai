# Story 8.3: Cost Display Per Role & Config Template Update

Status: done

## Story

As a developer reviewing run costs,
I want cost breakdowns split by model role (generation cost vs. review cost),
so that I can see exactly how much each phase of the pipeline costs and optimize model selection accordingly.

## Acceptance Criteria (BDD)

1. **Given** cost data accumulated in `BudgetState` with per-invocation tracking **When** `arcwright-ai status` displays cost information **Then** it includes a cost breakdown by role: "Generation cost: $X (N invocations)" and "Review cost: $Y (M invocations)" in addition to the existing total cost display.

2. **Given** a completed or in-progress run **When** the run summary (`summary.md`) is generated **Then** it includes a "Cost by Model Role" section with a markdown table showing role, model version, invocations, tokens (in/out), and cost for each configured role.

3. **Given** per-story cost data in the budget **When** the per-story cost table is displayed in both status and summary **Then** it distinguishes between generation and review costs when both are present.

4. **Given** `BudgetState` or `StoryCost` in `core/types.py` **When** per-role cost tracking is needed **Then** a `cost_by_role: dict[str, Decimal]` field is added to `StoryCost` (default empty dict) to support the breakdown without requiring schema-breaking changes. The `BudgetState` aggregate fields (`estimated_cost`, `total_tokens`, etc.) remain unchanged — they still track totals.

5. **Given** the `_serialize_budget()` function in `output/run_manager.py` **When** it serializes `BudgetState` **Then** it correctly serializes the new per-role `cost_by_role` fields from `StoryCost` to `run.yaml`, converting `Decimal` values to `str`.

6. **Given** `arcwright-ai init` is run **When** the default config template is generated **Then** it uses the new `models` format (with `generate` and `review` sections) instead of the old `model` format.

7. **Given** the config template generated by `init` **When** a developer reads it **Then** it includes inline comments explaining model role configuration, available roles, and how fallback from `review` to `generate` works.

8. **Given** cost formatting in both CLI and summary **When** costs and tokens are displayed **Then** formatting remains human-readable: "$1.17" not "0.00117 USD", "12,450 tokens" not "12450".

9. **Given** a run executed with only the `generate` role configured (no explicit `review`) **When** cost display functions are called **Then** all costs show under "Generation" only, with no empty "Review" row or role breakdown — backward compatible display.

10. **Given** story implementation is complete **When** `ruff check .` is run against the FULL repository **Then** zero violations.

11. **Given** story implementation is complete **When** `.venv/bin/python -m mypy --strict src/` is run **Then** zero errors.

12. **Given** all public classes and functions **When** every docstring is inspected **Then** each conforms to Google-style (Args, Returns, Raises sections where appropriate).

13. **Given** any existing tests in the full test suite **When** this story is complete **Then** no existing tests break. All 829 existing tests continue to pass.

14. **Given** unit tests **When** the test suite runs **Then** tests verify: (a) per-role cost display formatting, (b) summary table generation with role breakdown, (c) backward compat — runs with old single-model data display correctly without role breakdown, (d) config template generates valid YAML with `models` section, (e) `_serialize_budget()` handles `cost_by_role` Decimal serialization.

## Boundary Conditions

- **Boundary 1 (No review role in budget data):** When a run only used the `generate` role (no validation occurred, or validation fell back to generate), the `cost_by_role` dict will only have a `"generate"` key. Display functions MUST handle this gracefully — show only the "Generation" row, no empty "Review" row.
- **Boundary 2 (Empty `cost_by_role`):** Runs executed before this story (old `run.yaml` files) will have `StoryCost` entries without `cost_by_role`. The serialization and display functions MUST handle missing/empty `cost_by_role` dicts by falling back to showing total cost only (no role breakdown).
- **Boundary 3 (StoryCost backward compat):** The `cost_by_role` field uses `default_factory=dict` making it optional. All existing code that constructs `StoryCost` continues to work without modification. The `BudgetState.per_story` values retain their existing fields — `cost_by_role` is purely additive.
- **Boundary 4 (`_serialize_budget` Decimal handling):** The `cost_by_role` dict values are `Decimal`. The serializer must iterate `cost_by_role` and convert each value to `str`, similar to how `estimated_cost` and `max_cost` are already handled.
- **Boundary 5 (Config template — no api section):** The updated config template must NOT include an `api:` section. API keys are set via environment variables or global config only (`~/.arcwright-ai/config.yaml`). The existing comment at the top of the template already explains this — preserve it.
- **Boundary 6 (Summary Cost by Role table aggregation):** The "Cost by Model Role" table in `summary.md` must AGGREGATE across all stories. For each role, sum the `cost_by_role[role]` across all `per_story` entries. This is computed at display time, not stored in `BudgetState`.
- **Boundary 7 (Engine nodes — budget accumulation):** `agent_dispatch_node` in `engine/nodes.py` must update `cost_by_role` when constructing `StoryCost`: add the invocation cost under the `"generate"` key. `validate_node` must add validation cost under the `"review"` key. Both use the same `model_copy(update={...})` frozen-model pattern already established.
- **Boundary 8 (Per-story table role split):** When both `generate` and `review` costs exist for a story, the per-story table should show separate "Gen Cost" and "Review Cost" columns (or equivalent). When only one role exists, those columns show "$0.00" for the absent role.

```python
class StoryCost(ArcwrightModel):
    """Per-story cost breakdown for a single story within a run.

    Tracks token consumption and cost for all invocations (first-pass +
    retries) of a single story.  Frozen per ``ArcwrightModel`` convention;
    accumulate via ``model_copy(update={...})``.

    Attributes:
        tokens_input: Total input tokens consumed for this story.
        tokens_output: Total output tokens consumed for this story.
        cost: Estimated cost in USD (exact Decimal).
        invocations: Number of SDK invocations for this story.
        cost_by_role: Per-role cost breakdown mapping role name
            (e.g. ``"generate"``, ``"review"``) to cost in USD (Decimal).
    """

    tokens_input: int = 0
    tokens_output: int = 0
    cost: Decimal = Decimal("0")
    invocations: int = 0
    cost_by_role: dict[str, Decimal] = Field(default_factory=dict)
```

**In `cli/status.py` — Role cost aggregation helper:**
```python
def _aggregate_role_costs(per_story: dict[str, Any]) -> dict[str, dict[str, Any]]:
    """Aggregate cost_by_role across all stories.

    Returns dict keyed by role name with {"cost": Decimal, "invocations": int}.
    """
    roles: dict[str, dict[str, Any]] = {}
    for _slug, entry in per_story.items():
        cost_by_role = entry.get("cost_by_role", {}) or {}
        for role, cost_val in cost_by_role.items():
            if role not in roles:
                roles[role] = {"cost": Decimal("0"), "invocations": 0}
            roles[role]["cost"] += Decimal(str(cost_val))
            roles[role]["invocations"] += 1
    return roles
```
