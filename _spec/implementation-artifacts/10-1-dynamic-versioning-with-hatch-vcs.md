# Story 10.1: Dynamic Versioning with hatch-vcs

Status: done

## Story

As a maintainer of Arcwright AI,
I want the package version to be derived automatically from git tags using hatch-vcs,
So that version management requires zero manual edits, dev builds are uniquely identifiable, and releases are a simple `git tag` + push.

## Acceptance Criteria (BDD)

1. **Given** `pyproject.toml` exists with a static `version = "0.1.0"` **When** the migration is applied **Then** the `version` key is removed from `[project]`, `"version"` is added to the `dynamic` list, and `[tool.hatch.version]` configures `hatch-vcs` as the version source

2. **Given** `hatch-vcs` is configured in `pyproject.toml` **When** `[build-system]` is inspected **Then** `requires` includes both `"hatchling"` and `"hatch-vcs"` so the build backend can resolve the version at build time

3. **Given** `src/arcwright_ai/__init__.py` currently has `__version__ = "0.1.0"` hardcoded **When** the migration is applied **Then** the version is read dynamically at runtime via `importlib.metadata.version("arcwright-ai")` with a fallback to `"0.0.0.dev0"` for editable installs without a git tag

4. **Given** a `_version.py` file is generated by hatch-vcs at build time **When** `[tool.hatch.version]` is inspected **Then** it includes `raw-options.version_scheme = "guess-next-dev"` and `raw-options.local_scheme = "no-local-version"` for clean PEP 440 versions in CI/PyPI

5. **Given** the repository has no version tags yet **When** `git tag v0.1.0` is created on the current HEAD **Then** `pip install -e .` produces a package reporting version `0.1.0`, and `arcwright-ai --version` (if version flag exists) or `python -c "import arcwright_ai; print(arcwright_ai.__version__)"` outputs `0.1.0`

6. **Given** the tag `v0.1.0` exists and 3 commits have been made since **When** building the package **Then** the version resolves to something like `0.1.1.dev3` (PEP 440 dev version), uniquely identifying the dev build

7. **Given** the `[dev]` optional dependencies **When** inspected **Then** `hatch-vcs` is NOT listed as a dev dependency (it is a build dependency in `[build-system].requires` only — not needed at runtime or in dev environments)

8. **Given** the migration is complete **When** `ruff check src/` and `mypy --strict src/` are run **Then** zero violations and zero type errors

9. **Given** the migration is complete **When** `pytest` is run **Then** all existing tests pass with no regressions

10. **Given** this is a build infrastructure change **When** reviewing CI implications **Then** `.github/workflows/ci.yml` requires no changes (hatch-vcs resolves version from `.git` automatically during build, which CI already has)

## Dev Notes

**Why hatch-vcs:**
- Industry standard for Hatchling projects (equivalent to `setuptools-scm` for setuptools)
- Git tags as single source of truth — no version string to maintain
- Dev builds auto-tagged with distance from last tag (e.g., `0.1.1.dev3`)
- PEP 440 compliant out of the box
- Used by pytest, pip, Ruff, and other major Python projects

**Version scheme choices:**
- `guess-next-dev` — after `v0.1.0` tag + 3 commits → `0.1.1.dev3` (bumps patch, adds dev suffix)
- `no-local-version` — omits `+gabcdef1` local identifier, which PyPI rejects

**Release workflow going forward:**
1. Make changes, merge PRs
2. When ready to release: `git tag -a v0.X.Y -m "v0.X.Y — description"`
3. `git push origin v0.X.Y`
4. CI builds + publishes to PyPI (when CI publish workflow is configured)

**Rollback:** If hatch-vcs causes issues, revert to static `version = "X.Y.Z"` in pyproject.toml and hardcoded `__version__` in `__init__.py`. No other code depends on the versioning mechanism.
