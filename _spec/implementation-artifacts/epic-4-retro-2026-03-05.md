## What Went Well

**Output package built clean with perfect package boundaries.** Three new modules (`provenance.py`, `run_manager.py`, `summary.py`) all depend only on `core/` — zero engine imports. The dependency DAG constraint `output → core` held firm across all four stories. Each module was independently testable without standing up the LangGraph engine.

**`finalize_node` design routes elegantly.** Story 4.4's architectural decision to add a single `finalize_node` that all terminal paths route through (budget exceeded, escalated, success) is clean and testable. Alternative approaches (per-terminal summary, CLI post-hook) were explicitly considered and rejected in the story spec. The resulting graph shape is symmetric and easy to reason about.

**Best-effort artifact writing consistently applied.** Every output API call in Story 4.4 is wrapped in `try/except Exception` with WARNING-level logging and continuation. No artifact write can halt orchestration. This pattern was in the story AC and carried through to all four wiring calls — the product now handles I/O failures gracefully at the right boundary.

**`commit_node` placeholder debt from Epics 2–3 resolved.** The placeholder node that passed state through unchanged since Story 2.1 is now a real implementation calling `update_story_status()` and `update_run_status()`. This was a 2-epic-old debt item now cleared.

**Story 4.3 review cycle completed in one revision round.** The initial implementation had 1 HIGH and 3 MEDIUM findings. All were corrected in a single revision. The HIGH finding (resume placeholder formatting) was an AC-level miss with a testable, well-bounded fix. The review process worked as designed — caught meaningful issues, made them actionable, got to done.

**V6 backtick bug caught and fixed same day via real-world use.** When running `dispatch --story 28.4` against the market-matrix project, a false-positive V6 escalation was caught. Diagnosis, fix, regression test, and commit took under 30 minutes. The fix (`.strip("\`")` on path candidates in `_extract_file_paths`) is minimal and sound.

**LangGraph Studio setup unlocked as bonus.** `langgraph.json` config added, `.venv-studio` environment established with Python 3.13 (required due to `pyo3` wheel constraints on Python 3.14), and Studio verified working. Graph visualization now available at `https://smith.langchain.com/studio/?baseUrl=http://127.0.0.1:2024`.

## What Could Be Improved

**V6 backtick bug not caught by tests — real agent output patterns under-covered.** The `_extract_file_paths` regex test corpus only included `## File: path.py` headers and fenced code blocks. The agent's actual output used inline code formatting in headers (`### \`path/file.py\` — description`). This pattern wasn't in any test case. The fix is committed and regression-tested, but it raises a question: are there other agent output patterns the V6 checker doesn't handle?

**Story 4.4 specification created retroactively.** The implementation was committed (as unstaged changes) before the story file existed. This is a process inversion — spec should precede implementation, not follow it. The story file was created in the same commit as the implementation files, which means the spec could not have informed the implementation. The implementation appears correct, but the process deviation means: (a) the dev agent had no story spec to reference, (b) the story AC and pitfalls blocks were written after-the-fact rather than guiding development.

**File list discrepancy continues (systemic — still no tooling fix).** The Epic 3 retro identified this as a tooling gap at 7/11 stories across two epics. Actions 3 and 4 remain deferred. The pattern is still present: the git commit "feat(story-4.4)" included implementation files that were staged implicitly via `git add -A` rather than a deliberate file-by-file reconciliation. Without an automated diff check, this will keep happening.

**Python 3.14 + LangGraph Studio incompatibility creates two-venv friction.** The project uses Python 3.14 for development (`.venv`) but `langgraph-api`'s `pyo3`-based dependencies don't publish 3.14 wheels. This requires maintaining a separate `.venv-studio` on Python 3.13. The README documents this, but it adds ongoing maintenance overhead every time Studio dependencies need updating.

**Story 4.3 review found resume placeholder formatting as AC-level defect.** The initial `write_timeout_summary()` implementation generated `EPIC-<EPIC>` for unparseable slugs instead of the graceful fallback behavior specified in AC #7. This is the kind of AC-level miss that self-contained ACs are meant to prevent (Action 4 from Epic 3). It suggests AC precision is still variable across stories — some ACs are implementation-complete, others leave ambiguity.

## Key Insights

1. **Real-world dispatch surfaced a bug that 372+ tests didn't.** The V6 backtick issue only manifested with actual agent output. This suggests test corpus coverage should include realistic agent output samples — not just synthetic minimal inputs. One strategy: capture sanitized agent outputs from real runs and use them as integration test fixtures.

2. **Process inversions (impl before spec) are hard to detect and correct after the fact.** When an implementation is committed without a story file, the spec written afterward is descriptive rather than prescriptive. The story 4.4 spec is comprehensive, but its ACs were written to match what was implemented, not to drive what should be implemented. The `create-story` workflow should be a gating requirement, not retroactive documentation.

3. **Two deferred action items (3 and 4 from Epic 3) are now three epics old.** Actions 3 (git diff audit) and 4 (automated quality gates) were raised in Epic 2's retro, re-raised in Epic 3's retro, and remain open. Each retro defers them to "tooling investment not yet made." At some point, the recurring cost of manual work and review findings exceeds the investment cost. That point may have arrived.

4. **LangGraph Studio adds real value for graph comprehension.** Seeing the graph visually — six nodes, three terminal paths, retry loop — communicates the architecture faster than any prose. This should be documented as part of new contributor onboarding.

5. **The output package is a complete, stable foundation.** `provenance.py`, `run_manager.py`, and `summary.py` are all implemented, tested, and wired. Epic 5 can consume them directly. The run artifact trail is production-grade.

## Key Technical Patterns Established in Epic 4

1. **Best-effort artifact writes** — `try: await output_fn(...) except Exception: logger.warning(...)` — every output call in engine nodes is non-blocking; orchestration never halts on I/O failure
2. **`finalize_node` terminal convergence** — all terminal graph paths (success, escalated, budget exceeded) converge at a single finalize node before END; summary writing is centralized
3. **`append_entry()` idempotent composition** — provenance entries are inserted before the `## Validation History` section; the function reads, modifies, and rewrites — safe for sequential writes in the same async context
4. **`_derive_halt_reason()` from state inference** — halt reason (v6_invariant_failure, max_retries_exhausted, budget_exceeded) derived from `state.retry_history` and `state.retry_count` — not stored as an explicit field
5. **Three-function summary pattern** — `write_success_summary`, `write_halt_report`, `write_timeout_summary` are separate functions with distinct signatures — not a parametric dispatcher. Each reads `run.yaml` independently via `get_run_status()`.
6. **`update_story_status()` / `update_run_status()` two-level tracking** — story-level (status, timestamps, retry_count) and run-level (last_completed_story, budget) updated separately in `commit_node`
7. **Run ID format: `YYYYMMDD-HHMMSS-<4-char-id>`** — time-sortable, collision-resistant, human-readable in directory listings
8. **`.venv-studio` Python 3.13 env** — separate from main `.venv` (Python 3.14); required because `pyo3`-based packages (`langgraph-api`) have no 3.14 wheel and can't be compiled without matching `pyo3` support
