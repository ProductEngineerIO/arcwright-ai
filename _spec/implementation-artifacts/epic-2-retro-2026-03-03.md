## What Could Be Improved

**Dev Agent Record file lists not matching git reality (4/7 stories).** Stories 2.2, 2.3, 2.6, and 2.7 all had review findings where the recorded file list didn't match actual git changes. Root cause: the dev agent doesn't have a mechanism to reconcile its file list with git state before finalizing the story record. This undermines provenance and makes reviews harder.

**Exit code / terminal state mapping bugs (2/7 stories).** Story 2.1 had the budget-exceeded flow not marking escalated state. Story 2.7 returned `EXIT_SUCCESS` even when final state was `ESCALATED`, and the epic dispatch loop treated any non-exception return as success. Root cause: the happy path is easy to test; failure-exit mapping is easy to forget. ACs described exit codes but didn't mandate test assertions for each code path.

**Integration tests break on placeholder-to-real transitions (2/7 stories).** Stories 2.6 and 2.7 both broke existing graph integration tests when placeholder nodes became real I/O. Each required creating/updating fixtures (`graph_project_state`, `mock_agent`). This will recur in Epic 3 when `validate_node` becomes real. The pattern is predictable — it should be front-loaded into story specs.

**Initial implementations missing documented required features (2/7 stories).** Story 2.2 shipped without `§` architecture anchor support. Story 2.3 shipped without the `_patterns` precompiled regex map. Both were explicitly in ACs. Root cause: story specs are thorough (400+ lines), and ACs that reference implementation details via indirection to dev notes can get lost. ACs need to be self-contained.

**Shared lint/check command still not implemented (carried from Epic 1).** Action 2 from Epic 1 retro was not addressed. Low severity but reflects a commitment gap.

**Logger level side-effect leakage (Story 2.7).** `_setup_run_logging` set `arcwright_ai` logger to DEBUG without restoring previous level. Caught in review, but symptomatic of "side effects in setup functions" pattern.

## Key Insights

1. **Story spec pitfalls blocks are proven.** The front-loaded pitfalls block from Epic 1 retro eliminated 100% of recurring issues. This is the primary mechanism for cross-story learning in a stateless agent system. Continue and expand.

2. **Exit code assertions must be mandatory, not descriptive.** ACs should not just say "exit code 3 on failure" — they should mandate a test assertion for each exit code path. The gap between specification and verification is where bugs hide.

3. **Placeholder-to-real transitions are predictable.** Every time a placeholder node becomes real, integration tests that exercise the graph will break. This is knowable at story creation time. Front-load it.

4. **ACs must be self-contained.** If an AC requires a specific implementation detail (like a `_patterns` regex map or `§` anchor support), the AC itself must describe it — not point to a dev notes section. Indirection creates opportunities for omission.

5. **Canonical test fixtures compound in value.** `MockSDKClient` was defined once in Story 2.5 and reused without modification across 2.6 and 2.7. Investing in good test infrastructure early pays dividends. Story 3.2 will need to extend it for multi-call sequences.

## Key Technical Patterns Established in Epic 2

1. `model_copy(update={...})` — universal state update pattern for Pydantic models in graph nodes
2. `@dataclass(frozen=True)` — internal data objects that don't need Pydantic overhead
3. Module-level compiled regex constants (`_FR_PATTERN`, `_NFR_PATTERN`, `_ARCH_PATTERN`, etc.)
4. Async-first I/O — `read_text_async()`, `write_text_async()`, `asyncio.to_thread(mkdir)`
5. Node error pattern: log entry → catch → log error + exit → re-raise
6. Dependency injection via `@runtime_checkable` Protocol (`PathValidator`)
7. Lazy SDK imports for testability via monkeypatching
8. `MockSDKClient` as canonical SDK test fixture
9. Defense-in-depth: dual sandbox enforcement (SDK `can_use_tool` hook + post-stream validation)
10. Structured logging: `logger.info("event.name", extra={"data": {...}})`
